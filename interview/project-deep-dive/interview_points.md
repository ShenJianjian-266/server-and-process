# 项目面试考察点

## 项目一句话

这是一个面向车辆采集数据的数据管理与解析平台：Java 服务负责数据登记、状态流转、接口和 Kafka 消息衔接；Python 服务负责从 S3 拉取 MCAP、视频、点云、结构化数据，完成解析、切片、压缩、抽稀、上报和回传。

## 1. 数据上传后的端到端链路设计

### 概述

考察候选人是否能讲清楚一条数据从上传消息进入系统，到解析完成并可查询的完整链路。

### 具体论述

该项目不是单纯 CRUD。Java 端监听 `sacp-dms-data-upload`，接收上传消息后创建数据主记录，再发送 `sacp-dms-data-parse-start` 触发 Python 解析。Python 端根据 S3 路径下载原始数据，完成分类、校验、可视化生成、结果上报，最后回写数据状态。

这个点值得深挖，因为它天然包含“系统边界”：上传方、管理端、解析端、S3、Kafka、数据库之间如何协作。候选人如果只能说“收到消息后处理文件”，说明理解较浅；如果能说清楚状态流转、失败重试、重复消息、数据幂等和结果回写，说明真正理解了业务闭环。

### 可追问方向

- Kafka 消息消费成功后为什么手动 ack？
- 创建数据记录成功但发送解析消息失败怎么办？
- Python 解析成功但状态回写失败怎么办？
- 如何保证同一个数据不会被重复解析？
- 如果 S3 文件还没完全上传就触发了解析，如何处理？

## 2. Kafka 消息驱动与幂等控制

### 概述

考察候选人对异步系统可靠性的理解，而不是只会使用 Kafka 注解。

### 具体论述

Java 端消费上传消息时，会根据 `dataName` 判断数据是否已存在。重复数据会 ack 并跳过，避免同一条消息反复重试造成阻塞。普通异常则不 ack，让 Kafka 触发重试。

这个设计背后有几个可深挖点：哪些错误应该重试，哪些错误不应该重试；业务唯一键怎么选；重复消息是否一定无害；数据库写入和 Kafka 发送之间是否存在分布式一致性问题。

面试官通常会对这个点感兴趣，因为它能区分“写过消息消费代码”和“真正处理过生产消息异常”的候选人。

### 可追问方向

- `dataName` 作为幂等键是否足够？
- 如果数据库插入成功，但解析启动消息发送失败，会出现什么问题？
- 是否需要 outbox 表或本地消息表？
- 消费失败一直重试会不会造成分区阻塞？
- 如何设计死信队列和告警？

## 3. 多模态数据解析：MCAP、视频、点云、结构化数据

### 概述

考察候选人是否理解该项目的业务技术难点：它处理的是自动驾驶采集数据，不是普通文本或表单数据。

### 具体论述

Python 解析端要处理多类数据：结构化 topic、点云、视频、geo、truth 等。普通数据处理关注字段转换，而这里还要处理 MCAP 文件完整性、topic 分类、时间戳范围、点云坐标转换、视频转码、可视化文件生成。

例如点云处理不仅是“读出来再写回去”，还涉及 CDR 反序列化、voxel 抽稀、cm 到 m 坐标转换、序列化写回。视频处理也不是普通文件复制，而是围绕 H265、分片上下文、降采样、重压缩进行处理。

这个点适合深挖候选人的工程复杂度认知：是否知道数据格式、时间戳、编码压缩、文件损坏、跨模态对齐都会影响最终结果。

### 可追问方向

- 为什么点云可视化要抽稀？
- 点云抽稀会不会影响精度，如何取舍？
- 视频分片处理为什么可能依赖前一个分片？
- MCAP 文件损坏时为什么不一定直接失败？
- 如何判断解析结果是正确的？

## 4. 数据状态机与失败语义

### 概述

考察候选人是否能把“状态字段”讲成业务状态机，而不是简单枚举。

### 具体论述

项目中数据状态包含已上报、处理中、解析完成、解析失败等阶段。Java 端创建主记录时设置初始状态；Python 开始解析时更新处理中；解析成功或失败后再回写最终状态。

难点在于失败不只有一种。例如 MCAP 尾部文件损坏，可能是车端异常关闭导致，系统可以跳过末尾损坏文件继续解析；但中间文件损坏虽然也可跳过，最终可能需要标记异常，避免业务方误以为数据完整。

这类状态语义很适合面试深挖，因为它反映候选人是否理解“系统对外承诺了什么”。解析完成不一定代表数据质量完全正常，解析失败也不一定代表没有任何产物。

### 可追问方向

- 解析失败后已上传的部分结果怎么处理？
- 是否需要“部分成功”状态？
- 状态更新是否允许回退？
- 如何避免并发更新导致状态覆盖？
- 失败原因应该放在哪张表、保留多久？

## 5. 文件完整性校验与异常恢复

### 概述

考察候选人面对不稳定原始数据时的容错设计能力。

### 具体论述

Python 流水线会在解析前检查 MCAP 文件能否读取 summary。发现损坏文件后，不是简单中断，而是区分末尾损坏和中间损坏，并通过重命名 `.broken` 跳过异常文件。

这体现了真实数据平台的工程判断：采集端可能因为断电、网络、异常关闭生成不完整文件。如果平台只要遇到坏文件就失败，会降低可用性；如果完全忽略坏文件，又会让数据质量不可控。

合理做法是“尽量产出 + 明确标记 + 可追溯”。这个考察点能引出数据质量、异常降级、可追溯审计、重跑机制等话题。

### 可追问方向

- 为什么尾部损坏和中间损坏处理策略不同？
- `.broken` 重命名是否会影响后续重跑？
- 如果校验本身很慢，如何优化？
- 是否需要记录损坏文件清单？
- 如何让业务方知道数据不完整？

## 6. 性能优化：并发、I/O、单遍处理

### 概述

考察候选人是否做过有数据量压力的处理，而不是只写功能。

### 具体论述

项目中有多个性能优化点。事件解析中，D0/D1/D2/D3/D4/D8 按依赖关系组织，D2 结构化数据先产出 checkpoint，之后视频、点云、truth 可以并发处理。普通数据解析中，点云可视化按切片并发处理，并将“抽稀 + 坐标转换”合并为单遍读取、单次序列化，减少中间文件 I/O。

这类优化不是单纯加线程，而是先识别依赖关系和瓶颈：S3 下载是 I/O 密集，视频转码可能是 CPU/外部进程密集，点云解析涉及反序列化和序列化。不同瓶颈适合不同并发策略。

这个点面试价值高，因为面试官可以追问“为什么这样并发”“怎么证明变快”“会不会把机器打满”。

### 可追问方向

- 线程数如何确定？
- 点云处理为什么要减少一次读写？
- 如何定位瓶颈是 CPU、磁盘还是网络？
- 大文件处理如何控制本地磁盘占用？
- 并发失败时如何回收部分结果？

## 7. 时间戳体系与跨模态对齐

### 概述

考察候选人是否理解车辆数据中“时间”不是一个简单字段。

### 具体论述

项目中存在 PTP 时间、车端时间、文件时间、事件时间等多个时间概念。Python 流水线会从 MCAP 中提取 PTP 时间范围和 veh 时间范围，还会计算 `veh_ptp_offset`，用于后续视频、点云、事件片段的对齐。

这是自动驾驶数据平台的核心问题之一。视频帧、点云帧、结构化信号如果不能对齐，后续回放、质检、标注、算法分析都会受到影响。

普通候选人可能只会说“取开始时间和结束时间”；更好的候选人会意识到不同设备时钟、采样频率、延迟、丢帧、时区格式都会影响对齐结果。

### 可追问方向

- PTP 时间和车端时间有什么区别？
- `veh_ptp_offset` 怎么计算，失败时怎么办？
- 视频帧和点云帧如何按时间对齐？
- 时间戳缺失或乱序如何处理？
- 展示给用户的时间为什么要转北京时间？

## 8. 数据上报接口与批量写入设计

### 概述

考察候选人是否理解解析结果如何落库，以及批量接口的必要性。

### 具体论述

Python 端不会只生成文件，还会调用 Java 管理端接口批量创建切片主记录、原始切片记录、可视化记录等。比如 struct 切片会先创建主记录并拿到 `sliceId` 映射，后续点云、视频、geo 等结果再挂到对应切片上。

这个点的核心是“解析产物之间有关联关系”。如果每个文件单独上报，接口调用多、性能差，也容易出现部分成功导致数据不一致。批量上报能减少请求次数，但也要求服务端处理好事务、校验和返回明细映射。

### 可追问方向

- 批量创建部分失败怎么办？
- `sliceId` 映射丢失会影响哪些后续流程？
- 是否应该由服务端生成所有切片关系？
- 接口调用失败是否重试，重试是否幂等？
- 上报文件和上传 S3 的顺序如何设计？

## 9. 本地工作目录与资源清理

### 概述

考察候选人是否关注长时间运行服务的资源管理。

### 具体论述

Python 服务会把 S3 数据下载到本地工作目录，解析过程中生成切片、可视化文件、中间文件。处理完成后根据配置清理工作目录；事件解析在调试模式下保留目录，方便复用。

这个点看似简单，但生产环境中非常关键。视频和点云文件通常很大，如果不及时清理，本地磁盘会快速打满；如果清理太激进，又会影响失败排查和重跑。

好的回答应包含：工作目录隔离、按 dataId/eventInfoId 命名、异常情况下的 finally 清理、调试模式保留、磁盘用量监控、上传成功后再删除本地文件。

### 可追问方向

- 如果服务中途崩溃，残留目录如何清理？
- 清理动作如何避免误删？
- 磁盘不足时任务应该排队还是失败？
- 是否需要按任务隔离临时目录？
- 如何支持失败任务重跑？

## 10. 可观测性：traceId、阶段日志、耗时统计

### 概述

考察候选人是否具备生产排障意识。

### 具体论述

项目中 Python 日志支持 traceId，流水线按 Step 输出阶段日志，并记录点云处理耗时、事件各阶段耗时。Java 端也有请求日志、TraceId 拦截器和 Kafka 消费日志。

对于这种多服务、多文件、多阶段的数据处理系统，日志不是辅助功能，而是定位问题的主要工具。一次任务可能跨 Kafka、Java、Python、S3、数据库，如果没有统一标识和阶段耗时，很难判断卡在下载、解码、上传还是状态回写。

这个点容易引出更高级的话题：日志结构化、指标监控、任务追踪、慢任务告警、错误分类、链路追踪。

### 可追问方向

- traceId 如何贯穿 Java 和 Python？
- 如何定位一个任务卡住在哪一步？
- 哪些指标应该上报到监控系统？
- 如何区分业务失败和系统异常？
- 日志中如何避免泄露 S3 密钥等敏感信息？

## 11. 配置与部署边界

### 概述

考察候选人是否理解项目从本地运行到生产部署需要哪些配置边界。

### 具体论述

Java 服务依赖 Spring Boot、Kafka、数据库、Nacos、Liquibase；Python 服务依赖 FFmpeg、S3、Kafka、外部 API、第三方点云工具。两端都提供 Dockerfile 和 build 脚本。

这个点不是问“会不会写 Dockerfile”，而是看候选人是否知道哪些东西应该配置化，哪些不能写死，哪些依赖必须在镜像或宿主机中准备好。例如 FFmpeg 缺失会导致视频处理失败；S3、Kafka、数据库凭据不能进入代码仓库；不同环境需要不同 `application-*.yml`。

### 可追问方向

- FFmpeg 应该放在镜像里还是宿主机？
- Nacos 配置和本地 yml 如何分工？
- Liquibase 在生产环境如何执行？
- 镜像版本如何和代码版本对应？
- 密钥如何注入，如何避免泄露？

## 12. 项目亮点总结

这个项目适合在面试中重点讲三条主线：

1. 业务闭环：上传消息进入系统，生成主记录，触发解析，产物上报，状态闭环。
2. 数据复杂度：处理 MCAP、视频、点云、结构化、truth 等多模态车辆数据。
3. 工程可靠性：Kafka 幂等、异常恢复、批量上报、并发优化、traceId 日志、资源清理。

如果需要把它压缩成面试表达，可以这样说：

> 我负责的是车辆采集数据从上传到解析完成的后端链路。Java 服务负责数据登记、状态管理和 Kafka 触发；Python 服务负责从 S3 下载 MCAP、视频、点云等数据，做完整性校验、切片、点云抽稀、视频转码和结果上报。项目里比较值得展开的是消息幂等、解析失败恢复、多模态时间对齐、并发处理和大文件资源清理。
