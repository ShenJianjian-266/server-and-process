# 任务回调讲解

## 1. 一句话理解

“任务回调”可以理解为：**Argo 任务跑完或状态变化后，主动通知我们这个后端系统**。

不是我们一直去问 Argo：“你跑完了吗？你成功了吗？”而是 Argo 在关键时刻主动发一个 HTTP 请求给我们：“我这个任务现在成功了、失败了、结束了。”

换句话说：

> 我们把任务交给 Argo 执行，Argo 执行完后，再调用我们项目里的一个接口，把执行结果告诉我们。

## 2. 回调接口长什么样

后端可以提供一个接口，例如：

```http
POST /argo/tasks/callback
```

Argo 任务结束后调用这个接口，传入类似这样的数据：

```json
{
  "workflowName": "data-process-adc-12345-x7k9p",
  "status": "Succeeded",
  "message": "任务执行成功",
  "finishedAt": "2026-07-07 15:30:00"
}
```

后端收到后，根据 `workflowName` 找到数据库里的任务记录，把状态从 `SUBMITTED` 或 `RUNNING` 更新成 `SUCCEEDED`。

## 3. 为什么需要任务回调

现在项目的问题是：

1. 后端调用 `ArgoWorkflowClient.submit(...)`。
2. Argo 创建了一个 Workflow。
3. 后端拿到 `workflowName`。
4. 但后端不知道这个 Workflow 后来怎么样了。

任务后续可能处于这些状态：

- 正在执行
- 执行成功
- 执行失败
- 被取消
- 超时

如果没有回调，Web 页面就没法准确展示任务状态，也没法判断哪些任务可以重跑。

所以导师说的“任务回调”，本质是为了让我们系统知道：

> Argo 任务最终执行结果是什么。

## 4. 完整流程

```text
1. 用户或 Kafka 触发业务
        ↓
2. 后端调用 ArgoWorkflowClient 提交 Argo 任务
        ↓
3. Argo 返回 workflowName
        ↓
4. 后端保存一条任务记录，状态 = SUBMITTED
        ↓
5. Argo 开始执行任务
        ↓
6. Argo 任务结束
        ↓
7. Argo 调用我们后端的回调接口
        ↓
8. 后端更新任务状态 = SUCCEEDED / FAILED
        ↓
9. Web 页面查询任务列表，看到最新状态
```

## 5. 本项目里的例子

原始数据上传后，`DataUploadConsumer` 会调用：

```java
argoWorkflowClient.submitDataProcess(...)
```

这个方法最终会提交一个 Argo 数据解析任务。

假设 Argo 返回的任务名是：

```text
data-process-adc-20567100000098-abcd1
```

那我们后端应该先保存一条记录：

```text
workflowName = data-process-adc-20567100000098-abcd1
taskType = DATA_PROCESS
dataId = 20567100000098
status = SUBMITTED
```

等 Argo 跑完后，它回调我们：

```json
{
  "workflowName": "data-process-adc-20567100000098-abcd1",
  "status": "Failed",
  "message": "S3 file not found"
}
```

我们系统收到后更新数据库：

```text
status = FAILED
message = S3 file not found
finishTime = 当前时间
```

这样前端任务管理页面就能展示：

```text
任务名：data-process-adc-20567100000098-abcd1
原始数据ID：20567100000098
状态：失败
失败原因：S3 file not found
操作：重新解析
```

## 6. 回调和轮询的区别

| 方式 | 含义 | 特点 |
| --- | --- | --- |
| 回调 | Argo 主动通知我们 | 实时、效率高 |
| 轮询 | 我们定时去问 Argo 状态 | 简单但浪费资源，可能有延迟 |

导师说“加回调”，意思是更希望用第一种：

```text
Argo 主动调我们
```

而不是：

```text
我们每隔几秒查一次 Argo
```

## 7. 我们项目里大概要做什么

为了支持任务回调，后端大概要做这些事：

1. 新增 Argo 任务表，保存每次提交记录。
2. 改造 `ArgoWorkflowClient` 或调用方，提交成功后保存 `workflowName`。
3. 新增回调接口，例如：

```java
@PostMapping("/argo/tasks/callback")
public CommonResult<Boolean> callback(@RequestBody ArgoTaskCallbackDTO dto) {
    argoTaskService.handleCallback(dto);
    return CommonResult.success(true);
}
```

4. Argo WorkflowTemplate 中配置任务结束后调用这个接口。
5. 回调接口收到结果后更新任务状态。
6. 页面查询任务表，展示状态。
7. 失败任务支持重新提交。

## 8. 新手最容易混淆的点

`workflowName` 很关键。

它是连接 Argo 和我们系统的桥梁：

```text
提交任务时：Argo 返回 workflowName
回调任务时：Argo 带回 workflowName
后端更新时：用 workflowName 找到任务记录
```

所以如果没有保存 `workflowName`，后面 Argo 回调来了，我们就不知道它对应的是哪条业务记录。

## 9. 最终要理解的核心

导师说的“任务回调”不是一个复杂概念，它就是：

> Argo 任务执行完后，主动调用我们后端接口，把任务结果告诉我们；我们再把结果保存到数据库，供任务管理页面展示和重跑使用。
