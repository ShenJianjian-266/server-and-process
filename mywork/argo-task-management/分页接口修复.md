# Argo 任务分页接口修复方案

## 1. 文档目标

本文档用于修复 Argo 任务分页接口中分页元数据不准确的问题。

按照本文操作后，接口 `POST /argo/tasks/page` 应正确返回：

- 当前页码 `pageNum`
- 每页条数 `pageSize`
- 数据库总记录数 `total`
- 总页数 `pages`
- 当前页数据 `list`
- 上一页、下一页等导航信息

本文档给出了可以直接复制使用的业务代码、单元测试代码和验证命令。整个修复只需要改一个业务方法并新增一个测试类，不需要修改 Controller、Mapper、Mapper XML、数据库或 Maven 依赖。

## 2. 问题现象

前端请求第二页：

```json
{
  "pageNum": 2,
  "pageSize": 10
}
```

假设数据库中实际有 25 条符合条件的任务，正确结果应该是：

```json
{
  "pageNum": 2,
  "pageSize": 10,
  "total": 25,
  "pages": 3,
  "list": [
    "第 11 至第 20 条数据"
  ]
}
```

但当前接口会出现类似结果：

```json
{
  "pageNum": 1,
  "pageSize": 10,
  "total": 10,
  "pages": 1,
  "list": [
    "第 11 至第 20 条数据"
  ]
}
```

可以看到，`list` 中的数据确实是第二页的数据，但 `pageNum`、`total` 和 `pages` 都不正确。这会导致前端分页器无法显示正确的总条数，也无法正常切换页码。

## 3. 当前代码在哪里

需要修改的文件：

```text
sacp-dms-data-service/src/main/java/com/seres/sacp/dms/module/argo/service/impl/ArgoTaskServiceImpl.java
```

当前 `page` 方法如下：

```java
@Override
public PageInfo<ArgoTaskVO> page(ArgoTaskQuery query) {
    PageHelper.startPage(query.getPageNum(), query.getPageSize());
    List<ArgoTask> list = argoTaskMapper.selectPageList(query);
    return new PageInfo<>(list.stream().map(this::toVO).toList());
}
```

问题出在最后一行：

```java
list.stream().map(this::toVO).toList()
```

## 4. 新手先理解 PageHelper 如何工作

### 4.1 `PageHelper.startPage` 做了什么

执行下面这行代码时：

```java
PageHelper.startPage(2, 10);
```

PageHelper 会记录“下一次 MyBatis 查询需要查第 2 页，每页 10 条”。紧接着执行 Mapper 查询时，PageHelper 会自动完成两件事：

1. 执行 count SQL，得到符合条件的总记录数，例如 25。
2. 给原查询添加分页限制，查询第 11 至第 20 条数据。

因此下面的 `list` 虽然声明类型是 `List<ArgoTask>`，运行时通常实际是 PageHelper 的 `Page<ArgoTask>`：

```java
List<ArgoTask> list = argoTaskMapper.selectPageList(query);
```

`Page` 实现了 `List` 接口，所以可以赋值给 `List`。但它比普通 List 多保存了这些分页信息：

```text
pageNum = 2
pageSize = 10
total = 25
pages = 3
```

### 4.2 为什么执行 Stream 后元数据会丢失

当前代码执行：

```java
list.stream().map(this::toVO).toList()
```

这一步会创建一个新的普通 List。新 List 中只有转换后的 `ArgoTaskVO` 数据，不再是 PageHelper 的 `Page`，所以 `total`、`pageNum`、`pageSize`、`pages` 等信息全部丢失。

接下来再执行：

```java
new PageInfo<>(普通List)
```

`PageInfo` 无法知道数据库原来有 25 条数据，只能把这个普通 List 当成完整结果，于是推断：

```text
pageNum = 1
pageSize = 当前 List 的大小
total = 当前 List 的大小
pages = 1
```

这就是分页元数据不准确的根本原因。

## 5. 修复思路

修复需要严格遵守这个顺序：

```text
Mapper 返回带分页信息的 Page
            ↓
先用它创建 PageInfo<ArgoTask>
            ↓
再调用 PageInfo.convert 将实体转换为 VO
            ↓
得到 PageInfo<ArgoTaskVO>
```

当前项目使用的 PageHelper 版本支持：

```java
PageInfo<E> convert(Page.Function<T, E> function)
```

`convert` 会做两件事：

1. 把 `list` 中的每个 `ArgoTask` 转换成 `ArgoTaskVO`。
2. 把原来的 `total`、`pageNum`、`pageSize`、`pages` 和导航字段完整复制到新的 `PageInfo`。

所以不需要手工复制分页字段，也不需要再执行一次 count 查询。

## 6. 可直接使用的修复代码

打开：

```text
sacp-dms-data-service/src/main/java/com/seres/sacp/dms/module/argo/service/impl/ArgoTaskServiceImpl.java
```

将原来的 `page` 方法完整替换为：

```java
/**
 * 分页查询 Argo 任务列表。
 *
 * @param query 查询条件和分页参数
 * @return 保留完整分页元数据的 Argo 任务分页结果
 */
@Override
public PageInfo<ArgoTaskVO> page(ArgoTaskQuery query) {
    // PageHelper 只会拦截 startPage 后紧接着执行的第一条 MyBatis 查询。
    PageHelper.startPage(query.getPageNum(), query.getPageSize());

    // 该 List 在运行时实际是 Page<ArgoTask>，内部保存了 total、pageNum、pages 等信息。
    List<ArgoTask> entityList = argoTaskMapper.selectPageList(query);

    // 必须先构造 PageInfo，保留 Page 中的分页元数据。
    PageInfo<ArgoTask> entityPageInfo = new PageInfo<>(entityList);

    // convert 只转换列表元素，同时会复制全部分页元数据。
    return entityPageInfo.convert(this::toVO);
}
```

也可以写成更紧凑的形式：

```java
@Override
public PageInfo<ArgoTaskVO> page(ArgoTaskQuery query) {
    PageHelper.startPage(query.getPageNum(), query.getPageSize());
    List<ArgoTask> entityList = argoTaskMapper.selectPageList(query);
    return new PageInfo<>(entityList).convert(this::toVO);
}
```

建议新手使用第一种写法，因为每一步的类型和目的更清楚。

### 6.1 Import 是否需要修改

当前类已经导入了以下类型：

```java
import com.github.pagehelper.PageHelper;
import com.github.pagehelper.PageInfo;
import java.util.List;
```

因此替换方法后不需要增加新的 import。

### 6.2 不需要修改的文件

以下文件不需要修改：

```text
ArgoTaskController.java
ArgoTaskService.java
ArgoTaskMapper.java
ArgoTaskMapper.xml
ArgoTaskQuery.java
ArgoTaskVO.java
pom.xml
```

Mapper XML 中不要手写 `LIMIT`，也不要单独新增 count SQL。PageHelper 会自动处理分页 SQL 和 count SQL。

## 7. 新增单元测试

建议新增文件：

```text
sacp-dms-data-service/src/test/java/com/seres/sacp/dms/module/argo/service/impl/ArgoTaskServiceImplTest.java
```

完整测试代码如下，可以直接使用：

```java
package com.seres.sacp.dms.module.argo.service.impl;

import com.github.pagehelper.Page;
import com.github.pagehelper.PageHelper;
import com.github.pagehelper.PageInfo;
import com.seres.sacp.dms.client.ArgoWorkflowClient;
import com.seres.sacp.dms.config.ArgoProperties;
import com.seres.sacp.dms.module.argo.entity.ArgoTask;
import com.seres.sacp.dms.module.argo.enums.ArgoTaskStatusEnum;
import com.seres.sacp.dms.module.argo.enums.ArgoTaskTypeEnum;
import com.seres.sacp.dms.module.argo.mapper.ArgoTaskMapper;
import com.seres.sacp.dms.module.argo.query.ArgoTaskQuery;
import com.seres.sacp.dms.module.argo.vo.ArgoTaskVO;
import org.junit.jupiter.api.AfterEach;
import org.junit.jupiter.api.Test;
import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;
import org.mockito.Mock;
import org.mockito.junit.jupiter.MockitoExtension;

import static org.assertj.core.api.Assertions.assertThat;
import static org.mockito.Mockito.verify;
import static org.mockito.Mockito.when;

@ExtendWith(MockitoExtension.class)
class ArgoTaskServiceImplTest {

    @Mock
    private ArgoProperties argoProperties;

    @Mock
    private ArgoWorkflowClient argoWorkflowClient;

    @Mock
    private ArgoTaskMapper argoTaskMapper;

    @InjectMocks
    private ArgoTaskServiceImpl argoTaskService;

    @AfterEach
    void clearPageHelper() {
        // 单元测试中 Mapper 是 mock，不会触发 MyBatis 插件自动清理分页上下文。
        // 主动清理可以避免本测试影响其他测试。
        PageHelper.clearPage();
    }

    @Test
    void page_shouldKeepPaginationMetadataAndConvertEntityToVO() {
        // 1. 模拟前端请求第 2 页，每页 10 条。
        ArgoTaskQuery query = new ArgoTaskQuery();
        query.setPageNum(2);
        query.setPageSize(10);
        query.setTaskType(ArgoTaskTypeEnum.DATA_PROCESS.getCode());

        // 2. 准备一条当前页数据。
        ArgoTask task = new ArgoTask();
        task.setId(1001L);
        task.setWorkflowName("data-process-1001-abcde");
        task.setTemplateName("sacp-dms-data-process-with-qc");
        task.setTaskType(ArgoTaskTypeEnum.DATA_PROCESS.getCode());
        task.setStatus(ArgoTaskStatusEnum.FAILED.getCode());
        task.setParameters("{\"data_id\":\"1001\"}");

        // 3. Page 是 PageHelper 查询结果的真实运行时类型。
        //    这里模拟数据库总共有 25 条，当前请求第 2 页。
        Page<ArgoTask> mapperPage = new Page<>(2, 10);
        mapperPage.setTotal(25L);
        mapperPage.add(task);
        when(argoTaskMapper.selectPageList(query)).thenReturn(mapperPage);

        // 4. 调用待测试的方法。
        PageInfo<ArgoTaskVO> result = argoTaskService.page(query);

        // 5. 验证分页元数据没有在实体转 VO 时丢失。
        assertThat(result.getPageNum()).isEqualTo(2);
        assertThat(result.getPageSize()).isEqualTo(10);
        assertThat(result.getTotal()).isEqualTo(25L);
        assertThat(result.getPages()).isEqualTo(3);
        assertThat(result.isHasPreviousPage()).isTrue();
        assertThat(result.isHasNextPage()).isTrue();

        // 6. 验证列表元素已经从 ArgoTask 转换为 ArgoTaskVO。
        assertThat(result.getList()).hasSize(1);
        ArgoTaskVO taskVO = result.getList().getFirst();
        assertThat(taskVO.getId()).isEqualTo(1001L);
        assertThat(taskVO.getWorkflowName()).isEqualTo("data-process-1001-abcde");
        assertThat(taskVO.getStatus()).isEqualTo(ArgoTaskStatusEnum.FAILED.getCode());

        verify(argoTaskMapper).selectPageList(query);
    }
}
```

### 7.1 这个测试在验证什么

测试故意构造了以下场景：

```text
请求页码：2
每页条数：10
数据库总数：25
总页数：3
```

如果仍使用旧代码：

```java
return new PageInfo<>(list.stream().map(this::toVO).toList());
```

测试中的 `pageNum`、`total`、`pages` 断言会失败。

换成 `PageInfo.convert` 后，分页元数据被完整保留，所有断言通过。

### 7.2 为什么测试中要调用 `PageHelper.clearPage`

正常运行时，Mapper 会经过 MyBatis 和 PageHelper 拦截器，分页上下文会在查询结束后被清理。

这个单元测试使用 Mockito 模拟 Mapper，没有真正进入 MyBatis 拦截器，所以需要在 `@AfterEach` 中主动执行：

```java
PageHelper.clearPage();
```

否则当前测试调用的 `PageHelper.startPage` 可能把分页上下文留在线程中，影响后续测试。

## 9. 接口联调验证

代码测试通过后，启动服务并准备至少 11 条 Argo 任务数据，然后请求第二页。

本地环境请求示例：

```bash
curl -X POST 'http://localhost:20001/sacp-dms-data-server/argo/tasks/page' \
  -H 'Content-Type: application/json' \
  -d '{
    "pageNum": 2,
    "pageSize": 10
  }'
```

假设数据库中共有 25 条符合条件的任务，响应中的关键字段应类似：

```json
{
  "code": 0,
  "data": {
    "total": 25,
    "pageNum": 2,
    "pageSize": 10,
    "pages": 3,
    "hasPreviousPage": true,
    "hasNextPage": true,
    "list": [
      {
        "id": "2079410529841000449",
        "workflowName": "data-process-example",
        "status": "FAILED"
      }
    ]
  },
  "msg": ""
}
```

实际 `list` 数量取决于数据库数据。验证重点是：

```text
pageNum 等于请求的页码
pageSize 等于请求的每页条数
total 等于数据库符合筛选条件的总数
pages 等于向上取整后的总页数
```

总页数计算公式为：

```text
pages = 向上取整(total / pageSize)
```

例如：

```text
total = 25
pageSize = 10
pages = 3
```

## 10. 带筛选条件的验证

除了无条件分页，还要验证 count SQL 使用了相同筛选条件。

请求示例：

```bash
curl -X POST 'http://localhost:20001/sacp-dms-data-server/argo/tasks/page' \
  -H 'Content-Type: application/json' \
  -d '{
    "pageNum": 1,
    "pageSize": 10,
    "taskType": "DATA_PROCESS",
    "status": "FAILED",
    "workflowName": "data-process"
  }'
```

检查以下内容：

1. `list` 中的任务均符合筛选条件。
2. `total` 是符合筛选条件的总数，不是整张 `argo_task` 表的总数。
3. `pages` 根据筛选后的 `total` 计算。
4. 切换到第二页时，`pageNum` 正确返回 2。

## 11. 常见错误写法

### 11.1 错误：转换成普通 List 后再创建 PageInfo

```java
List<ArgoTaskVO> voList = entityList.stream()
        .map(this::toVO)
        .toList();
return new PageInfo<>(voList);
```

错误原因：`voList` 是普通 List，已经没有分页元数据。

### 11.2 错误：手工把 `total` 设置成当前页条数

```java
pageInfo.setTotal(voList.size());
```

错误原因：当前页只有 10 条不代表数据库总共只有 10 条。

### 11.3 错误：为了拿 total 再手写一次查询

```java
long total = argoTaskMapper.selectCount(...);
```

错误原因：PageHelper 已经自动执行 count SQL。再次查询会增加数据库开销，并且容易出现列表条件和 count 条件不一致。

### 11.4 错误：强制把 List 转成 Page

```java
Page<ArgoTask> page = (Page<ArgoTask>) entityList;
```

虽然 PageHelper 正常拦截时运行类型通常是 `Page`，但没有必要依赖强制类型转换。使用下面的写法已经可以识别并保留 Page 元数据：

```java
PageInfo<ArgoTask> pageInfo = new PageInfo<>(entityList);
```

### 11.5 错误：在 `startPage` 和 Mapper 查询之间执行其他查询

```java
PageHelper.startPage(pageNum, pageSize);
otherMapper.selectList();
List<ArgoTask> list = argoTaskMapper.selectPageList(query);
```

错误原因：PageHelper 默认分页紧接着执行的第一条 MyBatis 查询。上面的代码可能把分页应用到 `otherMapper.selectList()`，导致真正的任务查询没有分页。

正确写法：

```java
PageHelper.startPage(pageNum, pageSize);
List<ArgoTask> list = argoTaskMapper.selectPageList(query);
```

## 12. 修复后的数据流

```text
前端传 pageNum=2、pageSize=10
                ↓
PageHelper.startPage(2, 10)
                ↓
Mapper 查询，PageHelper 自动执行 count 和分页 SQL
                ↓
得到 Page<ArgoTask>
包含 total=25、pageNum=2、pageSize=10、pages=3
                ↓
new PageInfo<>(entityList)
保留分页元数据
                ↓
entityPageInfo.convert(this::toVO)
只把列表元素转换成 ArgoTaskVO
                ↓
返回 PageInfo<ArgoTaskVO>
total=25、pageNum=2、pageSize=10、pages=3 均保持正确
```
